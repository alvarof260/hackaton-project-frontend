
# Especificación de Requisitos de Software (SRS)

## Plataforma de Comisiones Artísticas

---

## 1. Introducción

### 1.1. Propósito

El propósito de este documento es detallar los requisitos del sistema propuesto para la gestión de comisiones artísticas. El sistema busca automatizar y transparentar la exhibición de catálogos, el cálculo de precios y el cobro de servicios de dibujo.

### 1.2. Ámbito del Sistema

El sistema busca digitalizar el proceso de encargo de ilustraciones. Permitirá a los artistas registrar catálogos con precios base, configurar calculadoras de extras, mostrar una galería de trabajos y gestionar una cola pública de proyectos paralelos. Además, integrará pasarelas de pago (Stripe, Mercado Pago, PayPal) para asegurar la transacción.

### 1.3. Definiciones, Acrónimos y Abreviaturas

| Término | Descripción |
|---------|-------------|
| **Catálogo** | Lista de servicios base que ofrece el artista (ej: Lineart, Full Color). |
| **Calculadora de Extras** | Herramienta que modifica el precio base según variables (ej: +1 personaje, uso comercial). |
| **Proyectos Paralelos (Queue)** | Lista pública que muestra el estado actual de los encargos en los que está trabajando el artista. |
| **Tier** | Nivel de servicio en el catálogo del artista. |
| **Addon** | Modificador de precio en la calculadora de extras. |
| **Comisión** | Pedido de un cliente a un artista por un trabajo artístico. |
| **Milestone** | Hito de pago (pago completo o seña + saldo). |

### 1.4. Referencias

Sin referencia.

## 2. Descripción General

### 2.1. Perspectiva del Producto

El sistema es una aplicación web (tipo SaaS/Marketplace) diseñada específicamente para artistas de dibujo que trabajan por comisiones. El objetivo principal es eliminar la fricción, los malos entendidos y las estafas en el proceso de contratación mediante:

- Estandarización de precios (catálogos y calculadoras)
- Transparencia del tiempo de espera (cola de proyectos paralelos)
- Seguridad financiera y legal mediante contratos y pagos integrados

### 2.2. Funciones del Producto

1. **Exhibición de Galería y Catálogo** — Portfolio visual y servicios ofrecidos.
2. **Cálculo de Precios Dinámico** — Calculadora de extras con actualización en tiempo real.
3. **Transparencia de Estado** — Cola pública de proyectos paralelos.
4. **Procesamiento de Pagos** — Integración con Mercado Pago, Stripe y PayPal.

### 2.3. Características de los Usuarios

| Rol | Descripción |
|-----|-------------|
| **Artista (Creador)** | Dueño de su tienda. Sube imágenes a su galería, define el catálogo base, configura la calculadora de extras y administra los estados de su cola de trabajo. |
| **Cliente (Comprador)** | Navega la galería, arma su cotización, acepta el contrato y gatilla el pago. |
| **Administrador** | Backoffice para gestionar reportes, resolver disputas y monitorear pasarelas de pago. |

### 2.4. Restricciones

- Compatible con cualquier dispositivo con acceso a internet.
- Depende de la disponibilidad de las APIs de Mercado Pago, Stripe y PayPal.
- Enfoque **Mobile First** (la mayoría accede desde redes sociales en celular).

### 2.5. Suposiciones y Dependencias

- Los artistas cuentan con cuentas activas en las pasarelas de pago mencionadas.
- Las imágenes públicas deben optimizarse (WebP) y servirse por CDN.

## 5. Flujo Core (Happy Path)

```
Cliente aterriza en perfil del Artista
         ↓
Mira la Galería para validar calidad
         ↓
Selecciona nivel de servicio del Catálogo (Boceto, Lineart, Full Color)
         ↓
Juega con la Calculadora de Extras y obtiene presupuesto exacto
         ↓
Manda el requerimiento detallado con referencias
         ↓
Artista revisa y hace clic en "Aceptar"
         ↓
Sistema genera el Acuerdo/Contrato automático
         ↓
Cliente paga (100% o 50% de seña) → Mercado Pago, Stripe o PayPal
         ↓
¡Trabajo entra automáticamente en Proyectos Paralelos públicos!
         ↓
Artista entrega obra final → Se libera pago restante → Ciclo cerrado
```

1. Planteamiento del Problema (Problem Statement)

Vender arte por encargo en 2026 sigue siendo un caos organizativo para la gran mayoría de los artistas independientes. A pesar de que la demanda de comisiones (para avatares, assets de streaming, ilustraciones personales o comerciales) está en su punto más alto, el flujo de trabajo está roto.

El problema real no es dibujar; es todo lo que pasa antes y después del dibujo.

Un artista promedio hoy sufre de:

    Fricción en la cotización: Mensajes interminables por Instagram o X (Twitter) del tipo "Hola, ¿cuánto cobrás por un dibujo de mi personaje con una espada y fondo de bosque?". El artista tiene que calcular mentalmente, explicar sus precios y rogar que el cliente no se asuste y desaparezca.

    Ansiedad del cliente (El síndrome "¿Para cuándo está?"): Como no hay un sistema de seguimiento transparente, el cliente manda mensajes constantemente interrumpiendo el flujo de trabajo del artista para pedir actualizaciones.

    Gestión de pagos manual: Depender de links sueltos de PayPal o Mercado Pago, revisar comprobantes a mano y el miedo constante al "chargeback" (desconocimiento de compra) o a entregar el arte y que no te paguen el saldo restante.

    Desorganización de los "Slots": Anunciar "Comisiones Abiertas (3/5 slots)" en un tweet que a las 2 horas queda sepultado en el feed, generando sobreventa o pérdida de oportunidades.

La mayoría de los artistas resuelve esto usando un frankenstein de herramientas: un link en bio (Linktree), un Google Docs o Carrd para los precios, Trello público para la cola de trabajo, y DMs para negociar.

Nuestro sistema elimina este frankenstein. Unificamos el catálogo, la calculadora de precios dinámica, la cola de trabajo pública y la pasarela de pagos en una sola herramienta diseñada específicamente para este nicho.
2. Visión del Producto

CommiFlow es el sistema operativo para el artista de comisiones.

No somos una simple vidriera. Somos una plataforma SaaS (Software as a Service) que profesionaliza el trato entre el creador y su cliente, brindando seguridad y transparencia.

Antes: "Mirá mis precios en este Google Doc, pasame las referencias por DM, te aviso cuando empiece tu dibujo y te paso el alias de Mercado Pago."

Después: El cliente entra al perfil del artista, elige el estilo, tilda "+1 personaje" y "Fast Delivery" en la calculadora, ve el precio exacto, sube sus imágenes de referencia, paga la seña y automáticamente ve cómo su nombre aparece en la cola de trabajo pública del artista en estado "En Espera". Es así de fácil, hermano.
3. Usuarios Objetivo (Target Users)
3.1. Primarios

    Artistas Freelance (Creadores): Ilustradores digitales, diseñadores de personajes, artistas de pixel art, riggers de VTubers. Buscan optimizar su tiempo, verse profesionales y asegurar su dinero.

    Clientes Frecuentes (Compradores): Creadores de contenido (streamers/youtubers), autores de rol (D&D), o fanáticos que buscan materializar sus ideas. Buscan un proceso de compra sin fricción, saber exactamente cuánto van a pagar y tener certeza de en qué estado está su pedido.

3.2. Secundarios

    Agencias de VTubing / Esports: Equipos que necesitan contratar arte de manera constante y requieren recibos claros y trazabilidad de los pedidos.

    Administradores del Sistema (Nosotros): Equipo de soporte técnico que monitorea la salud de la plataforma, resuelve errores y analiza métricas de conversión.

4. Arquitectura Técnica y Decisiones (Tech Stack)

Hermano, esta es la base de la catedral. Elegimos este stack por razones muy concretas de escalabilidad y mantenibilidad.
4.1. Frontend: React Puro (SPA)

    Por qué: Queremos interactividad inmediata. La Calculadora de Extras tiene que ser fluida; si el cliente tilda una opción, el precio total en pantalla debe actualizarse en 0 milisegundos sin recargar la página.

    Patrones a usar: * Vamos a manejar el estado global de la calculadora y el flujo de compra con un gestor liviano (ej: Zustand o Context API si es simple).

        React Router para la navegación entre el "Muro del Artista" y el "Tablero de Gestión".

        Trade-off asumido: Al no usar SSR (Next.js), sacrificamos el SEO dinámico de las tarjetas de Twitter/X por defecto, pero ganamos en simplicidad de despliegue y un frontend completamente desacoplado y cacheable en una CDN.

4.2. Backend: .NET 10 (C#)

    Por qué: Necesitamos un tipado fuerte, transacciones seguras (estamos manejando plata) y un rendimiento bestial. .NET 10 nos da las Minimal APIs que son ligerísimas para arrancar el proyecto, pero con toda la potencia del framework detrás para cuando necesitemos escalar.

    Patrones a usar:

        Arquitectura en Capas (Controllers/Minimal APIs -> Services -> Repositories).

        Entity Framework Core (EF Core) como ORM para mapear nuestros objetos C# a las tablas relacionales.

        Inyección de Dependencias (nativa en .NET) para mantener el código testeable.

4.3. Base de Datos: PostgreSQL

    Por qué: Relacional 100%. Las entidades de este negocio están fuertemente acopladas. Un Pedido pertenece a un Usuario (Cliente), requiere un Servicio (Tier), incluye múltiples Extras y genera un Pago. Postgres maneja estas relaciones, integridad referencial y bloqueos transaccionales mejor que nadie. Además, su soporte para JSONB nos va a salvar la vida si necesitamos guardar configuraciones dinámicas de la calculadora sin romper el esquema.

6. Requisitos Funcionales Detallados (RF)

Acá es donde bajamos la visión a tierra. Estos son los contratos de lo que el sistema DEBE hacer.
6.1. Módulo Core y Autenticación

    RF-01: Registro de Usuarios: El sistema debe permitir el registro de usuarios especificando su rol inicial: "Quiero ofrecer mis servicios (Artista)" o "Quiero encargar arte (Cliente)".

    RF-02: Autenticación JWT: El backend en .NET 10 debe proveer autenticación basada en JSON Web Tokens (JWT) para asegurar las rutas de las APIs.

    RF-03: Perfil Base: Todo usuario debe poder configurar un nombre de visualización, nombre de usuario (slug único para la URL, ej: /estanislao-art), avatar y biografía.

6.2. Módulo de Exhibición (Catálogo y Galería)

    RF-04: Gestión de Galería: El artista debe poder subir, eliminar y ordenar imágenes de muestra (formatos JPG, PNG, WebP) con un límite de peso por archivo.

    RF-05: Creación de Tiers (Servicios Base): El artista debe poder definir "Tiers" o niveles de servicio. Cada Tier debe contener:

        Título (Ej: "Busto Lineart").

        Descripción.

        Imagen de portada.

        Precio Base (Monto y Moneda, ej: ARS o USD).

        Tiempo Estimado de Entrega (ETA en días).

    RF-06: Configuración de Extras (Calculadora): El artista debe poder crear modificadores de precio aplicables a los Tiers. Tipos de Extras soportados:

        Suma Fija: Ej: "+ Arma detallada" (Suma $10).

        Multiplicador: Ej: "Uso Comercial" (Multiplica el subtotal x2).

6.3. Módulo de Compra (Frontend del Cliente)

    RF-07: Calculadora en Tiempo Real: El cliente, al visitar un Tier, debe ver un formulario dinámico en React. Al seleccionar extras, el "Precio Total Estimado" debe recalcularse instantáneamente en el cliente.

    RF-08: Formulario de Requerimientos: Antes de confirmar el pedido, el cliente debe proveer la información necesaria:

        Descripción de texto detallada.

        Subida de imágenes de referencia (hasta 5 imágenes).

    RF-09: Generación del Pedido: Al confirmar, el frontend debe enviar el payload completo (Tier, Extras seleccionados, Referencias) a la API de .NET, la cual creará la entidad Order con estado Pendiente de Aprobación.

6.4. Módulo de Gestión de Pedidos (El Tablero)

    RF-10: Bandeja de Entrada: El artista debe tener una vista en su dashboard con los pedidos entrantes (Pendiente de Aprobación).

    RF-11: Aceptación / Rechazo: El artista debe poder revisar las referencias del cliente y decidir Aceptar el pedido (lo que habilita el pago) o Rechazarlo (con un motivo opcional).

    RF-12: Estados del Proyecto: Una vez que el pedido está activo (pagado), el artista debe poder transicionar el estado del mismo:

        En Lista de Espera (Slot reservado).

        Bocetando (WIP 1).

        Lineart / Color (WIP 2).

        Entregado (Finalizado).

6.5. Módulo de Transparencia (La Cola / Queue)

    RF-13: Tablero Público de Cola: El "Muro" del artista debe mostrar públicamente una lista (read-only) de los pedidos activos y sus estados actuales.

    RF-14: Privacidad en la Cola: El sistema debe permitir al cliente elegir al momento de comprar si quiere que su nombre sea público en la cola de trabajo o figurar como "Cliente Anónimo".

6.6. Módulo de Pagos (Integración)

    RF-15: Configuración de Pasarela: El artista debe poder conectar sus credenciales de Mercado Pago (para moneda local) y/o Stripe/PayPal (para moneda extranjera) en su configuración.

    RF-16: Generación de Link de Pago: Cuando el artista Acepta un pedido (RF-11), el backend .NET debe comunicarse con la API de la pasarela correspondiente para generar un PaymentIntent o Link de Pago.

    RF-17: Webhooks de Confirmación: El backend .NET debe exponer endpoints seguros para recibir Webhooks de Mercado Pago/Stripe. Al recibir la confirmación de pago exitoso, el sistema debe cambiar automáticamente el estado del pedido de Esperando Pago a En Lista de Espera (activando el RF-13).

7. Modelo de Datos Conceptual (PostgreSQL + EF Core)

Para mapear esto con Entity Framework Core en .NET 10, vamos a usar un diseño altamente normalizado para el catálogo, pero desnormalizado e inmutable para los pedidos.

¿Por qué? Te lo explico como arquitecto: Si un artista hoy cobra un "Extra de Fondo" a $5000 y el mes que viene lo sube a $8000, los recibos de los pedidos viejos NO pueden cambiar. Por eso, en la orden guardamos una "foto" (snapshot) de los precios al momento de la compra.

Entidades Principales:

    Users (Usuarios Base):

        Id (UUID, PK)

        Username (String, Unique - Ej: 'estanislao_art')

        Email (String, Unique)

        PasswordHash (String)

        Role (Enum: Artist, Client, Admin)

        CreatedAt (DateTime UTC)

    ArtistProfiles (Extensión del Usuario):

        Id (UUID, PK)

        UserId (UUID, FK a Users, Unique)

        DisplayName (String)

        Bio (Text)

        IsQueueOpen (Boolean - Control maestro para pausar comisiones)

        MercadoPagoAccessToken (String, Encriptado)

        StripeAccountId (String)

    Tiers (El Catálogo de Servicios):

        Id (UUID, PK)

        ArtistId (UUID, FK a ArtistProfiles)

        Name (String - Ej: "Full Render")

        Description (Text)

        BasePrice (Decimal 18,2)

        Currency (String - Ej: "ARS", "USD")

        EstimatedDays (Int)

        IsActive (Boolean)

    Addons (Calculadora de Extras):

        Id (UUID, PK)

        TierId (UUID, FK a Tiers)

        Name (String - Ej: "+1 Personaje")

        Type (Enum: FixedAmount, PercentageMultiplier)

        Value (Decimal 18,2 - Ej: 15.00 o 1.5)

    Commissions (El Pedido / La Transacción Core):

        Id (UUID, PK)

        ClientId (UUID, FK a Users)

        ArtistId (UUID, FK a ArtistProfiles)

        Status (Enum: PendingApproval, AwaitingPayment, InQueue, WIP_Sketch, WIP_Color, Completed, Cancelled)

        ClientBrief (Text - Lo que pide el cliente)

        ReferenceImages (Array de Strings / JSON - URLs de AWS S3 o almacenamiento)

        FinalPrice (Decimal 18,2 - El cálculo final validado por el backend)

        SnapshotData (JSONB - FUNDAMENTAL: Acá guardamos cómo se componía el precio en ese momento exacto, ej: {"tierName": "Boceto", "basePrice": 1000, "addons": [{"name": "+Color", "price": 500}]}).

    Payments (Registro Financiero):

        Id (UUID, PK)

        CommissionId (UUID, FK a Commissions)

        Provider (Enum: MercadoPago, Stripe)

        ProviderTransactionId (String - El ID que nos devuelve la pasarela)

        Amount (Decimal 18,2)

        Status (Enum: Pending, Approved, Refunded)

8. Estrategia de Endpoints (.NET 10 Minimal APIs)

Hermano, en .NET 10 las Minimal APIs vuelan. No necesitamos controladores pesados, vamos directo al grano separando por dominios lógicos.

Dominio Público (Muro del Artista - Sin Autenticación)

    GET /api/v1/artists/{username} -> Devuelve Perfil + Estado de la cola.

    GET /api/v1/artists/{username}/tiers -> Devuelve el catálogo y los addons para armar la calculadora en React.

    GET /api/v1/artists/{username}/queue -> Devuelve la lista de Commissions activas (solo mostrando nombre del tier y estado, ocultando data sensible).

Dominio Transaccional (Cliente logueado)

    POST /api/v1/commissions -> El cliente envía el payload. El backend RECALCULA el precio para evitar que nos manipulen el frontend, guarda el SnapshotData y crea la orden en PendingApproval.

Dominio Backoffice (Artista logueado - Protegido con JWT + Rol)

    POST /api/v1/catalog/tiers -> Crea un nuevo servicio.

    PATCH /api/v1/dashboard/commissions/{id}/accept -> El artista acepta el pedido. Acción: Dispara la llamada a Mercado Pago/Stripe para generar el link de pago y se lo devuelve al cliente.

    PATCH /api/v1/dashboard/commissions/{id}/status -> Cambia de "Boceto" a "Finalizado".

Dominio de Integración (Webhooks)

    POST /api/v1/webhooks/mercadopago -> Endpoint seguro. Cuando Mercado Pago avisa que el cliente pagó, .NET busca la Commission, le cambia el estado a InQueue y libera el slot.

9. Requisitos No Funcionales (RNF)

Acá es donde separamos a los amateurs de los profesionales. Un sistema que maneja cobros tiene que ser una roca.

    RNF-01 (Rendimiento - Latencia): La carga del muro público del artista (GET /artists/{username}) debe resolverse en menos de 200ms. Si tenemos picos de tráfico (ej: el artista publica en Twitter), .NET debe cachear este endpoint (usando OutputCache nativo de .NET 10).

    RNF-02 (Seguridad - Zero Trust): BAJO NINGUNA CIRCUNSTANCIA la calculadora de React define el precio final a cobrar. El frontend solo estima para mostrarle al usuario. El cálculo real y absoluto se hace en el backend de .NET al momento de crear el POST /api/v1/commissions.

    RNF-03 (Seguridad - Manejo de Tarjetas): El sistema NO debe almacenar números de tarjeta de crédito ni CVVs. Todo el flujo de pago se delega (Off-site o vía SDK seguro) a Mercado Pago, Stripe o PayPal. Cumplimiento absoluto de normas PCI-DSS por delegación.

    RNF-04 (Almacenamiento de Archivos): Las imágenes de referencia (del cliente) y la galería (del artista) no se guardan en la base de datos ni en el servidor web. Se suben a un Object Storage (ej: AWS S3, Cloudflare R2) generando URLs prefirmadas (Presigned URLs) desde .NET para subidas directas, quitándole carga a nuestro backend.

    RNF-05 (Concurrencia - Race Conditions): Si un artista abre solo "2 Slots" y hay 50 personas intentando pagar al mismo tiempo, la base de datos PostgreSQL debe usar bloqueos transaccionales (Pessimistic Locking o Constraints) para asegurar que no se vendan 3 slots.

## 1. Problem Statement

Vender arte por encargo en 2026 sigue siendo un caos organizativo para artistas independientes. El flujo de trabajo actual está roto:

- **Fricción en la cotización**: Mensajes interminables por Instagram/X para presupuestar. El artista calcula mentalmente, explica precios, y reza para que el cliente no desaparezca.
- **Ansiedad del cliente**: Sin sistema de seguimiento transparente, el cliente interrumpe constantemente pidiendo actualizaciones.
- **Gestión de pagos manual**: Links sueltos de Mercado Pago, comprobantes revisados a mano, miedo constante al chargeback.
- **Desorganización de slots**: "Comisiones Abiertas (3/5)" en un tweet que a las 2 horas queda sepultado.

La mayoría de los artistas resuelve esto con un frankenstein de herramientas: Linktree, Google Docs para precios, Trello público para la cola, y DMs para negociar.

**CommiFlow elimina este frankenstein.** Unificamos catálogo, calculadora de precios dinámica, cola de trabajo pública y pasarela de pagos en una sola herramienta diseñada para este nicho.

### 6.1. Estados de Comisión (State Machine)

```
                    ┌─────────────────┐
                    │ PendingApproval │ ← Estado inicial al crear pedido
                    └────────┬────────┘
                             │
              ┌──────────────┼──────────────┐
              │              │              │
              ▼              │              ▼
     ┌────────────────┐      │     ┌───────────┐
     │ AwaitingPayment│      │     │ Cancelled │
     └───────┬────────┘      │     └───────────┘
             │               │     (artista rechaza)
             │               │
     (cliente paga)    (artista rechaza)
             │
             ▼
     ┌─────────┐
     │ InQueue │ ← Visible en cola pública
     └────┬────┘
          │
          ▼
     ┌─────────┐
     │   WIP   │ ← Artista está trabajando
     └────┬────┘
          │
          ▼
     ┌────────────┐
     │ Completed  │ ← Entregado
     └────────────┘
```

| Estado | Descripción | Quién transiciona |
|--------|-------------|-------------------|
| `PendingApproval` | Pedido creado, esperando revisión del artista | Sistema (al crear) |
| `AwaitingPayment` | Artista aceptó, esperando que el cliente pague | Artista (al aceptar) |
| `InQueue` | Pago confirmado, pedido en la cola pública | Sistema (webhook de pago) |
| `WIP` | Artista está trabajando en el pedido | Artista |
| `Completed` | Pedido entregado | Artista |
| `Cancelled` | Pedido rechazado o cancelado | Artista (rechazar) o Sistema (timeout futuro) |

**Transiciones válidas** (MUST validarse en backend):
- `PendingApproval` → `AwaitingPayment` (artista acepta)
- `PendingApproval` → `Cancelled` (artista rechaza)
- `AwaitingPayment` → `InQueue` (webhook confirma pago)
- `InQueue` → `WIP` (artista inicia trabajo)
- `WIP` → `Completed` (artista entrega)

Cualquier otra transición **MUST** ser rechazada por el backend con HTTP 409 Conflict.

## 7. API Endpoints

### 7.1. Público (Sin autenticación)

```
GET  /api/v1/artists/{username}
     → Perfil del artista + estado de la cola (IsQueueOpen)
     → Response: ArtistProfile DTO

GET  /api/v1/artists/{username}/tiers
     → Catálogo de servicios con sus addons
     → Response: List<TierWithAddonsDTO>

GET  /api/v1/artists/{username}/gallery
     → Imágenes del portfolio
     → Response: List<GalleryImageDTO>

GET  /api/v1/artists/{username}/queue
     → Cola de pedidos activos (solo tier name + estado, sin datos sensibles)
     → Response: List<QueueItemDTO>
     → SHOULD paginar con query params: ?page=1&pageSize=10
```

### 7.2. Cliente (Autenticado)

```
POST /api/v1/commissions
     → Crear pedido
     → Body: { tierId, addonIds[], clientBrief, referenceImageUrls[] }
     → Backend MUST recalcular el precio (NO confiar en el frontend)
     → Backend MUST guardar SnapshotData con la composición del precio
     → Response: CommissionDTO (con status: PendingApproval)
```

### 7.3. Artista (Autenticado + Rol Artist)

```
POST /api/v1/catalog/tiers
     → Crear tier
     → Body: { name, description, basePrice, estimatedDays }

PUT  /api/v1/catalog/tiers/{id}
     → Editar tier

DELETE /api/v1/catalog/tiers/{id}
     → Eliminar (soft delete → IsActive = false) o hard delete

POST /api/v1/catalog/tiers/{tierId}/addons
     → Crear addon para un tier

GET  /api/v1/dashboard/commissions
     → Bandeja de entrada (pedidos filtrados por estado)
     → SHOULD soportar filtro: ?status=PendingApproval

PATCH /api/v1/dashboard/commissions/{id}/accept
     → Aceptar pedido
     → Backend genera link de pago con SDK de MercadoPago
     → Response: { commissionId, paymentUrl, newStatus: "AwaitingPayment" }

PATCH /api/v1/dashboard/commissions/{id}/reject
     → Rechazar pedido
     → Body (OPTIONAL): { reason: "..." }
     → newStatus: "Cancelled"

PATCH /api/v1/dashboard/commissions/{id}/status
     → Cambiar estado del pedido
     → Body: { newStatus: "WIP" | "Completed" }
     → Backend MUST validar que la transición es válida (ver §6.1)
     → MUST retornar 409 si la transición no es válida
```

### 7.4. Webhooks (Sin autenticación por token, verificado por firma)

```
POST /api/v1/webhooks/mercadopago
     → MercadoPago envía notificación de pago
     → Backend MUST verificar firma HMAC del webhook
     → Al recibir "payment_approved":
       1. Buscar Commission por external_reference
       2. Validar que está en AwaitingPayment
       3. Crear registro en Payments
       4. Cambiar Commission status → InQueue
       5. Retornar HTTP 200
     → MUST retornar 200 rápido (< 5s) para que MercadoPago no reintente
```

## 8. Reglas de Negocio (Business Rules)

### BR-01: Recalculo de precio (CRITICAL)
El frontend calcula el precio para mostrar al usuario. El backend **MUST** recalcular el precio de forma independiente al recibir `POST /commissions`. Si el precio del frontend no coincide con el del backend (tolerancia: $0.01), el backend **MUST** usar SIEMPRE su propio cálculo y guardarlo en `FinalPrice` y `SnapshotData`.

### BR-02: Snapshot inmutable
Cuando se crea una Commission, `SnapshotData` **MUST** contener una copia exacta del nombre del tier, precio base, y cada addon con su nombre y precio al momento de la compra. Si el artista modifica los precios después, los pedidos existentes NO cambian.

### BR-03: IsQueueOpen
Si `ArtistProfile.IsQueueOpen = false`, el endpoint `POST /commissions` para ese artista **MUST** retornar HTTP 403 con mensaje "El artista no está aceptando comisiones actualmente".

### BR-04: Solo un pago activo por comisión
Si una Commission ya tiene un Payment en estado `Approved`, el backend **MUST NOT** generar un nuevo link de pago. Retornar HTTP 409.

### BR-05: Validación de transiciones
El backend **MUST** validar que cada cambio de estado de Commission siga la máquina de estados definida en §6.1. Transiciones inválidas **MUST** retornar HTTP 409.

## 16. Riesgos y Mitigaciones

| Riesgo | Probabilidad | Impacto | Mitigación |
|--------|-------------|---------|------------|
| SDK de MercadoPago tiene bugs/documentación confusa | Alta | Alto | Dedicar día 6 completo. Tener fallback de simular pago manualmente para la demo. |
| Upload de imágenes consume tiempo | Media | Medio | Usar upload local al VPS, no S3. Servir como static files. |
| .NET 10 tiene breaking changes o bugs | Baja | Alto | Usar features estables. Minimal APIs + EF Core son maduras. |
| CubePath deploy falla el día 7 | Media | Alto | Tener Docker Compose funcionando localmente ANTES del deploy. CubePath es solo "subirlo". |
| Scope creep (agregar features no planeadas) | Alta | Alto | Este documento es el contrato. Cualquier feature nueva pasa por revisión de scope. |

## 17. Decisiones Asumidas (Assumptions)

1. Moneda única: ARS (Peso Argentino) para todo el MVP.
2. Un solo rol por usuario (Artist o Client, no ambos).
3. No hay sistema de notificaciones. El estado se consulta activamente.
4. No hay paginación obligatoria en MVP (SHOULD, no MUST), pero los endpoints la soportan si hay tiempo.
5. Las imágenes se suben al mismo VPS, no a CDN externo.
6. No hay sistema de recuperación de contraseña para el MVP.
7. El webhook de MercadoPago no tiene retry robusto. Si falla, se procesa manualmente en la demo.
